# Offline Data Storage

## Обзор

При отключении устройства от интернета измерения не теряются: данные записываются во flash-память и отправляются на платформу при восстановлении соединения. Для корректной привязки измерений ко времени добавляется датчик RTC DS3231.

---

## Алгоритм записи

При каждой попытке отправки данных на сервер:
1. WiFi доступен → данные отправляются на сервер в компактном формате.
2. WiFi недоступен → данные записываются во flash в компактном формате.

Отдельного таймера для записи во flash нет — запись происходит в рамках штатного расписания передачи данных.

---

## Запись данных

- Пока интернет недоступен, данные записываются в OTA-раздел flash-памяти. В этот период OTA-раздел не нужен и не используется по своему прямому назначению.
- Записи пишутся последовательно, одна за другой.
- При включении устройство сканирует раздел с начала до первого пустого места и продолжает запись оттуда.
- При отключении питания данные в flash сохраняются.
- Если память OTA-раздела заполнена, новые записи не добавляются.

---

## Восстановление соединения

При появлении интернета:
1. Накопленные данные отправляются на платформу.
2. После успешной отправки данные стираются, OTA-раздел освобождается.
3. OTA-обновления разрешаются.

---

## Формат хранимых данных

Вместо полного набора полей измерений хранится только необходимое:

| Параметр  | Тип      | Точность                      |
|-----------|----------|-------------------------------|
| Timestamp | uint32_t | Unix-время (метка RTC DS3231) |
| CO2       | int16_t  | целые                         |
| Temp      | int16_t  | со знаком, десятые            |
| Humidity  | int16_t  | до десятых                    |
| TVOC      | int16_t  | до целых                      |
| PM2.5     | int16_t  | до десятых                    |

---

## Поведение WiFi

**Первый запуск / factory reset**
Устройство открывает точку доступа и ждёт 2 минуты. Если пользователь подключился — настраивает WiFi. Если никто не подключился — переходит в offline-режим.

**WiFi уже настроен, сеть доступна**
При включении устройство сразу подключается к сохранённой сети. Точка доступа не открывается. Накопленные offline-данные автоматически уходят на платформу. OTA разрешается.

**WiFi уже настроен, сеть недоступна**
Устройство работает в offline-режиме. Каждые 30 секунд по таймеру измерений проверяет доступность WiFi. Как только сеть появляется — автоматически подключается, отправляет данные, разрешает OTA.

---

## Аппаратное дополнение

Добавляется датчик RTC DS3231 для точной привязки каждого измерения ко времени при работе в офлайн-режиме. Время на датчике устанавливается сервисными специалистами на этапе установки прошивки.

---

## Ёмкость буфера

Размер одной записи: 14 байт (4 + 2 + 2 + 2 + 2 + 2).  
Размер OTA-раздела: 1 966 080 байт (~1.875 MB).  
Максимальное количество записей: ~140 000.

| Интервал передачи | Дней до заполнения |
|-------------------|--------------------|
| 1 минута          | ~97 дней           |
| 30 секунд         | ~48 дней           |

---

## Область применения

Фича реализуется только для WiFi-режима. Поддержка cellular-модуля (`A7672XX`) выпиливается: модуль физически не подключён, вся связанная логика удаляется из кодовой базы.

---

## Известные проблемы текущей версии

- Нет fallback на случай проблем с RTC модулем (разряженная батарея, модуль не установлен).
- Если раздел заполнен, новые измерения молча теряются.
- Не описан механизм отправки большого объёма накопленных данных (пачками или за раз).
- Не описан алгоритм поиска доступного места при старте. Линейный скан до первого «пустого» байта (`0xFF`) ненадёжен: валидные записи могут содержать `0xFF` в полях данных, а частично записанная запись при обрыве питания неотличима от свободного пространства. Формат маркера границы данных не определён.
- Не описано поведение при обрыве WiFi во время bulk-отправки накопленных данных. Часть записей уже отправлена на платформу, часть — нет. Стереть раздел нельзя (данные не полностью отправлены), не стереть — платформа получит дубли при следующей сессии. Механизм контрольной точки или идемпотентности отправки не определён.

## Возможности для расширения

### Дельта-кодирование для flash-буфера

Вместо хранения абсолютных значений каждого измерения хранить разности между соседними записями. Первая запись сохраняется полностью (14 байт), последующие — как дельты.

Типичные диапазоны изменений за 30 секунд:
- CO2: ±200 ppm → помещается в int8_t (±127) в большинстве случаев
- Temp: ±0.5°C (×10 = ±5) → int8_t
- Humidity: ±0.5% (×10 = ±5) → int8_t
- TVOC, PM2.5: аналогично

При кодировании дельт размер записи сокращается с 14 до ~5–6 байт. Ёмкость буфера вырастает в 2–3 раза без изменения железа (~280 000 записей вместо 140 000, ~194 дня при интервале 1 минута).

Открытые вопросы:
- Обработка выброса: если дельта не помещается в int8_t, нужен escape-механизм (например, маркер + полное значение)
- Формат записи: фиксированная длина упрощает поиск по индексу, переменная — сложнее навигация
- Точка синхронизации: при обрыве питания нужно знать, с какой абсолютной записи начинается цепочка дельт

### BLE как командный канал

Сейчас BLE используется только для WiFi-провижининга и останавливается после передачи credentials. В рамках расширения BLE-сервер можно использовать как командный канал для управления офлайн-буфером.

Возможные команды:
- Принудительная отправка накопленных данных
- Очистка буфера
- Чтение статуса буфера (количество записей, процент заполнения)

Открытые вопросы:
- Когда BLE-сервер активен: только в офлайн-режиме или всегда (влияет на энергопотребление)
- Авторизация: сейчас провижининг не защищён, для командного канала это риск
- Требует поддержки на стороне мобильного приложения
